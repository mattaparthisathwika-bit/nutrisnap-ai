## Quick start

Open a terminal in the app folder:

```powershell
cd d:\cal-ai-clone-starter\cal-ai-clone
```

```powershell
npm install
```

Then start the dev server:

```powershell
npm start
```

That opens the **Expo dev tools**. From there you can:

| Option | Command | How to use |
|--------|---------|------------|
| **Web** | `npm run web` | Opens in your browser (easiest on Windows) |
| **Android** | `npm run android` | Needs Android Studio emulator or a device |

Your `.env` is configured for demo mode, so food photo analysis uses built-in sample data — no HuggingFace key needed. You can try the app right away with `npm run web` or Expo Go.

## Real AI mode (optional)

To use actual HuggingFace vision instead of demo data:

1. Get a free API key from [HuggingFace tokens](https://huggingface.co/settings/tokens)
2. Add it to `.env`:
   ```
   EXPO_PUBLIC_HF_API_KEY=your_key_here
   ```
3. For **web**, run the proxy in a **second terminal** (avoids CORS):
   ```powershell
   npm run proxy
   ```
4. Keep `npm start` or `npm run web` running in the first terminal

**Cursor**

Investigating the profile save flow — checking the profile screen, context, and database layer.


Identifying likely causes: web `loadStore` can drop profile data on reload, native SQLite `UPDATE` may affect zero rows if no profile row exists, and the profile form resets from context. Fixing persistence and save flow.


There were **three bugs** causing profile save to fail or look like it failed:

## What was wrong

### 1. Web storage was losing profile data
On web, `loadStore()` used `...parsed` spread, which could **overwrite the saved profile** when reloading from `localStorage` (especially for older stored data without a `profile` key). Profile saves wrote correctly, but the next read could return defaults again.

### 2. SQLite `UPDATE` did nothing if no row existed
On mobile/native, `updateUserProfile` only ran `UPDATE ... WHERE id = 1`. If the `user_profile` row didn’t exist yet, **0 rows were updated** and nothing was saved (no error shown).

### 3. Form reset after save
A `useEffect` synced the form from `profile` on **every** context update. After save, if the reload returned stale data, your typed values were **wiped back** to defaults — making it look like save failed.

Also, on **web**, `Alert.alert()` often doesn’t show, so you might not have seen success or error messages.

## What we fixed

- **Web:** explicit `loadStore()` / `saveStore()` so `profile` is always preserved
- **Native:** upsert (`INSERT ... ON CONFLICT DO UPDATE`) + ensure profile row exists on init
- **Profile screen:** load form from storage only once on mount (won’t overwrite while you type)
- **Alerts:** use `window.alert` on web so you actually see “Saved” or errors

## What happened

When you pick a meal, the app saves the **full food photo** into browser `localStorage`. Photos are large (often several MB), but browsers only allow about **5MB total** for `localhost`. That’s why you saw:

> *"Could not save data. Storage may be full or blocked."*

## What we fixed

- Photos are now **compressed to a small thumbnail** before saving
- If storage is still too full, it **retries without large images** so the meal still saves

## What you should do

1. **Reload the app** — press `r` in the Expo terminal  
2. **Clear old bloated data** (one time, if it still fails):
   - Open browser DevTools (`F12`)
   - Go to **Application** → **Local Storage** → `http://localhost:8081`
   - Delete the key **`nutrisnap_web_store`**
   - Refresh the page

3. Scan again → pick **Lunch** (or any meal) → you should see **"Logged!"** and go to the Diary tab

The meal macros will save correctly; the diary may show a smaller thumbnail instead of the full-resolution photo.
